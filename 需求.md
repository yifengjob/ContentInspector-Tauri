## 一、规范遵循要求（重要：不得机械照搬 Electron 风格）

迁移时，目标项目必须遵循 **Tauri 官方最佳实践** 和 **Rust 生态规范**，不能将 Electron 项目中不适用于 Rust/Tauri 的内容机械照搬。具体要求如下：

### 1.1 命名规范
- **Rust 后端**：遵循 Rust 惯用命名规范
    - 变量、函数、模块、命令名：`snake_case`
    - 类型、结构体、枚举：`PascalCase`
    - 常量：`SCREAMING_SNAKE_CASE`
    - 示例：原 Electron 中的 `getConfigData` 应改名为 `get_config_data`（Rust 命令名），前端调用 `invoke('get_config_data')` 不变。
- **TypeScript 前端**：保持与原有项目一致（通常为 `camelCase` 函数、`PascalCase` 组件/类），但需注意前后端接口字段命名统一。
- **配置文件**：`tauri.conf.json` 中的键使用 `snake_case`（Tauri 标准）。

### 1.2 错误处理
- **Rust 后端**：使用 `Result<T, E>` 类型，自定义错误类型建议使用 `thiserror` 或 `anyhow`。不使用 `try-catch` 或 `throw`，不使用 `unwrap()`/`expect()`（仅限原型或确凿不变条件）。所有命令返回 `Result<T, Error>`，错误信息应清晰可读。
- **前端**：保留原有 `try-catch` 模式调用 `invoke`，正确处理 Rust 返回的错误。

### 1.3 异步编程
- **Rust 后端**：使用 `tokio` 异步运行时，遵循 `async`/`await` 最佳实践。避免长时间阻塞主线程，CPU 密集任务使用 `tokio::task::spawn_blocking`。不使用 `std::thread::sleep`，改用 `tokio::time::sleep`。
- **前端**：原有 `async/await` 保持不变，仅替换底层 API。

### 1.4 模块组织
- **Rust 后端**：遵循 Rust 模块系统（`mod.rs` 或分文件）。将不同功能拆分到独立模块（如 `commands/scheduler.rs`, `commands/extractors.rs`）。避免单一巨大 `main.rs`。
- **前端**：保留原项目的模块结构。

### 1.5 配置与权限
- 使用 `tauri.conf.json` 的 `allowlist` 声明所需权限（`fs`, `shell`, `dialog` 等），不得使用 `unsafe` 或绕过权限系统。
- 使用 Tauri 插件机制（如 `tauri-plugin-fs`、`tauri-plugin-shell`），而非直接调用 Rust 不安全 API。

### 1.6 依赖管理
- 使用 `crates.io` 上维护良好、安全的 crate，最小化依赖。优先使用 `tokio` 生态下的官方或广泛使用的 crate（如 `tokio`, `serde`, `anyhow`, `thiserror`, `num_cpus`, `sysinfo`）。
- 不直接引入 Node.js 绑定（除非绝对必要且无替代）。

### 1.7 安全性
- 遵循 Tauri 安全指南：不在前端使用危险操作，所有文件系统操作需经过 `allowlist` 配置，用户提供的路径需进行校验（防止路径遍历）。
- 不使用 `unsafe` 关键字，除非有充分理由并添加安全注释。

### 1.8 代码风格与工具
- 后端代码需通过 `rustfmt` 和 `clippy` 默认规则检查。
- 前端代码保持原有 ESLint/Prettier 配置（如有）。

### 1.9 跨平台兼容性
- 使用 `std::path::Path` 处理路径，避免硬编码分隔符（`/` 或 `\`）。
- 条件编译（`#[cfg(target_os)]`）仅在必要时使用。

**违反上述规范的实现将被视为不合格。**

## 二、保留所有优秀设计（前后端）

### 前端优秀设计
- 代码组织结构、组件拆分、路由设计、状态管理模式（原项目使用 Vue 3 + Pinia）与原项目保持一致。
- 样式组织、命名规范、错误处理前端部分、日志记录方式等完全继承。
- 仅将 Electron 特有交互（`ipcRenderer`、`window.electron`、`remote` 等）改为 Tauri API。

### 后端优秀设计（原 Electron 主进程）
- **并发模型**：原项目使用 Worker Threads 线程池（`worker-pool.ts`），CPU 密集型任务（`file-worker.ts`）与 I/O 密集型任务（`walker-worker.ts`）分离，支持异常自动重启。Tauri 中需用 Rust 的 `tokio::task::spawn_blocking` + 自定义线程池等价实现。
- **智能调度**：原项目实现 `smart-scheduler.ts`，采用 4 层调度策略（大文件动态并发、小文件自由并行、Worker 状态跟踪、任务优先级），并基于 CPU 核心数和可用内存动态调整并发数。Tauri 中必须完整复现调度算法。
- **流式文件处理**：原项目支持分块读取大文件 + 跨 chunk 关键词匹配（智能累积状态），支持 20+ 文件格式流式解析（PDF 逐页、Excel 流式等）。Tauri 中需使用 Rust 异步流 + 分块读取 + 状态结构体实现等价处理。
- **其他隐含设计**：缓存策略、重试机制、背压控制、资源生命周期管理、任务取消与进度上报等，也必须迁移到 Rust 后端。

## 五、Rust 后端实现要求（含技术细化）

### 5.1 并发处理架构

| 原 Electron 实现 | Tauri 迁移要求 |
|----------------|--------------|
| `worker-pool.ts`：Worker 线程池管理，支持生命周期与自动重启 | 使用 `tokio::task::spawn_blocking` 配合自定义线程池（`rayon` 或 `std::thread` + 通道），实现等价的生命周期管理与异常自动重启。创建 `ThreadPool` 结构体，管理线程创建、健康检查、重启逻辑。遵循 Rust 命名规范（`thread_pool`, `check_health`）。 |
| 双 Worker 类型分离：`file-worker.ts`（CPU 密集型）与 `walker-worker.ts`（I/O 密集型） | 使用不同优先级的执行器：CPU 任务放入 `spawn_blocking` 池（限制并发数），I/O 任务使用 Tokio 异步运行时直接 `spawn`。确保两者互不阻塞。 |
| 智能并发控制算法：基于 CPU 核心数、可用内存动态计算大文件并发数，小文件自由并行 | 使用 `tokio::sync::Semaphore` 实现动态信号量控制。在 Rust 命令中通过 `num_cpus::get()` 和 `sysinfo::System::new().available_memory()` 动态计算最大并发数。小文件任务通过异步批量投递保证高吞吐量，大文件使用专用信号量限制并发。 |
| 背压控制机制：基于系统负载自动调整任务投递速率 | 使用 Tokio 的 `mpsc::channel` 有界通道 + 异步等待，当通道满时自动阻塞生产者，实现等价背压控制。 |

### 5.2 智能调度策略

| 原 Electron 实现 | Tauri 迁移要求 |
|----------------|--------------|
| 4 层调度策略：大文件动态并发 + 小文件并行 + Worker 状态跟踪 + 任务优先级 | 实现调度器 Trait，管理任务优先级队列（使用 `std::collections::BinaryHeap` 配合自定义 `Task` 结构实现优先级）。大文件使用专用槽位（`Semaphore`），小文件使用通用池。低优先级任务在资源紧张时延迟执行。 |
| `smart-scheduler.ts` 统一管理扫描任务状态 | 创建 Rust 调度器模块（`scheduler.rs`），使用 `Arc<tokio::sync::Mutex<BinaryHeap<Task>>>` 管理任务队列。注意使用 `tokio::sync::Mutex` 避免 `await` 时的 Send 错误。 |
| 硬件配置自适应（CPU 核心数、可用内存） | 使用 `num_cpus` crate 获取核心数，`sysinfo` crate 获取内存信息，动态计算并发参数。支持在运行时根据剩余内存调整并发数。 |

### 5.3 流式文件处理

| 原 Electron 实现 | Tauri 迁移要求 |
|----------------|--------------|
| 分块读取大文件 + 跨 Chunk 匹配（智能累积状态） | 使用 `tokio::fs::File` + `tokio::io::AsyncReadExt::read` 分块读取（块大小 64KB）。实现搜索状态结构体 `MatchState`，存储部分匹配的关键词片段，跨越块边界时累积。匹配结果通过 Tauri Channel API 实时流式发送到前端。 |
| 20+ 文件格式支持（PDF 逐页解析、Excel 流式读取、Word 等） | - PDF 解析：使用 `pdf-extract` 或 `lopdf` 逐页读取流式处理。<br>- Excel 解析：使用 `calamine` 流式读取，支持 `.xls`, `.xlsx`。<br>- Word 解析：使用 `docx` crate（针对 `.docx`）或 `zip` + 自定义 XML 解析。<br>- 文本文件（TXT, CSV, JSON 等）：流式读取并实时处理。<br>- 其他格式（RTF, PPT 等）根据原项目支持列表按需实现。 |
| 提取器模式（统一接口 + 智能路由） | 在 Rust 中定义 `Extractor` trait（`async fn extract(&self, path: &Path, sender: &Channel<MatchResult>) -> Result<()>`）。实现 `TextExtractor`、`PdfExtractor`、`ExcelExtractor` 等。创建智能路由函数 `route_extraction`，根据文件扩展名动态调用对应提取器。 |
| 敏感数据实时识别与高亮 | 流式传输匹配结果时附带位置信息（行号、列索引、匹配文本）。前端按位置高亮。敏感类型包括：身份证号（正则 + 校验位）、手机号、银行卡号（Luhn 校验）、邮箱、IPv4 地址、密码关键词、中文姓名等。使用正则表达式 + 辅助校验函数。 |

### 5.4 任务取消与进度上报

| 原 Electron 实现 | Tauri 迁移要求 |
|----------------|--------------|
| 扫描任务支持取消 | 使用 `tokio_util::sync::CancellationToken` 在长时间运行的命令中定期检查 `is_cancelled()`。用户调用 `cancel_scan` 命令时触发取消。所有流式处理和调度循环必须检查取消标志。 |
| 实时扫描进度 | 在 Rust 命令中使用 `AppHandle::emit("scan-progress", ProgressData)` 发送进度事件（当前文件、已处理数量、总文件数、百分比）。前端通过 `listen` 监听并更新进度条。 |

### 5.5 需要实现的主要 Rust 命令与模块

| Electron 实现 | Rust 命令（及模块） |
|--------------|----------|
| `worker-pool.ts` + `smart-scheduler.ts` | `start_scan(state: State<Scheduler>, params: ScanParams) -> Result<String>`：创建调度器和任务队列，返回任务 ID。 |
| `smart-scheduler.ts` 中的取消 | `cancel_scan(state: State<Scheduler>, task_id: String) -> Result<()>`：通过 `CancellationToken` 取消指定任务。 |
| `config-manager.ts` | `get_config() -> Result<Config>` / `save_config(config: Config) -> Result<()>`：读写 `tauri::api::path::AppConfigDir` 下的配置文件（JSON）。 |
| `environment-check.ts` | `check_environment() -> Result<EnvironmentInfo>`：检测系统信息（OS 版本、可用内存、CPU 核心数等）。 |
| `extractors/*` | `parse_file(path: String, channel: Channel<MatchResult>) -> Result<()>`：根据文件类型流式解析并发送结果。 |
| `scan-state.ts` | 通过 `tauri::State<Arc<Mutex<Scheduler>>>` 管理全局任务调度器，`TaskManager` 管理进行中的任务。 |
| 报告导出（CSV/JSON/Excel） | `export_report(data: ReportData, path: String, format: String) -> Result<()>`：异步写入到用户选择的目标路径。 |

### 5.6 性能与可靠性要求

1. **Worker 异常恢复**：Worker 线程崩溃时自动重启，任务队列重新调度。—— 使用 `tokio::spawn` 包装带重试机制的 Future，或在线程池中监控 panic 并重建。
2. **内存占用控制**：单文件处理内存占用 < 200MB（与原项目一致），分块处理流式数据。
3. **扫描速度**：与原项目基准测试结果对比，差异应在 ±15% 以内。
4. **数据安全**：所有文件处理在本地完成，不上传任何内容。
